System Mission

Designed like a real telemetry pipeline

The simulator emits sensor data every second. The backend validates and stores it. The Python service detects abnormal values and generates reports. The desktop control panel gives operators a single place to view system state and alerts.

C++ -> Java API
Python -> Alerts + Report
C# -> Operator Dashboard
Assembly -> Checksum
VHDL -> Alert Signal Logic

End-to-End Flow

01

Generate telemetry

The C++ simulator creates temperature, pressure, vibration, oxygen, heart-rate, battery, timestamp, and checksum values every second.

02

Verify and store

The Java Spring Boot backend recomputes the checksum, accepts valid packets, and writes records into H2.

03

Detect abnormal values

The Python service reads records, applies threshold rules, writes a report, and posts alerts back to the backend.

04

Present to operator

The C# control panel shows the latest telemetry, recent alerts, monitoring state, and exported report output.

Core Components

C++ Sensor Simulator

Generates synthetic telemetry and sends packets to the Spring Boot backend once per second.

loop every 1 second
  build packet
  compute checksum
  POST /api/sensors

Java Backend API

Receives packets, verifies checksum integrity, stores telemetry in H2, and exposes alert endpoints.

verify checksum
save sensor record
serve latest values
store alerts

Python Analysis Service

Reads sensor data, detects abnormal readings, publishes alerts, and writes a report.

GET /api/sensors
apply thresholds
POST /api/alerts
write report

C# Control Panel

Provides operator controls for monitoring, status review, alert visibility, and report export.

start monitoring
stop monitoring
view alerts
export report

Assembly Checksum

Computes a byte-sum checksum for packet integrity before the backend accepts data.

xor eax, eax
add packet bytes
and eax, 0FFh
return checksum

VHDL Alert Circuit

Models hardware alert behavior where any danger input activates the final alert signal.

alert_signal <=
  temperature_high or
  pressure_high or
  oxygen_low or battery_low

What Makes It Realistic

  • One system produces telemetry while another validates integrity.
  • Analysis is separated from ingestion, which mirrors real monitoring architectures.
  • The operator view is distinct from the backend data plane.
  • The hardware alert model is represented separately in VHDL.
